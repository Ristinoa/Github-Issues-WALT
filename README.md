# Github-Issues-WALT
W.A.L.T. team Github public repository for Programming for Localization Week 8 group assignment.

## 📋 Project Task: Bug Safari & Triage Simulation

### **1. Issue Creation**
* **Quantity:** Create at least **5 issues** across your board.
* **Categories:** Use classifications from the "Bug Safari" exercise:
    * Text expansion (truncation)
    * Hardcoded strings (untranslated text)
    * RTL (Right-to-Left) layout issues
    * Date/Currency formatting
    * Contextual/Polysemy errors
* **Requirement:** Each issue must be a **proper bug report** (following Lesson 8.6 format), not just a title.


### **2. Assignment & Teamwork**
* **Distribution:** Assign at least one issue to **each group member**.
* **Accountability:** Every person on the team must have at least one issue assigned to them.


### **3. Active Response Workflow**
When a teammate assigns you an issue, you must:
* **Acknowledge:** Leave a comment (e.g., *"Looking into this"* or *"Need more info—what locale were you testing?"*).
* **Update Status:** Move the issue to **"In Progress."**
* **Resolve:** Add a follow-up comment with your resolution:
    * *Example:* "Fixed the truncation by shortening the German string."
    * *Example:* "This is NAB (Not A Bug)—the behavior matches the source language."
* **Finalize:** Move the issue to **"Done"** or reassign it back if more questions arise.


### **4. Lifecycle Management**
* **Visual Tracking:** The board should reflect multiple states (**To Do, In Progress, Done**) simultaneously.
* **Platform Specifics:**
    * **Bugzilla:** Change the Status field (**CONFIRMED ➡ IN_PROGRESS ➡ RESOLVED**).
    * **Jira/GitHub:** Drag issues across your Kanban columns.

> **Note:** The goal is to replace "Hey, can you do this?" emails with structured, trackable communication. Be mindful of "passive-aggressive" reassignment—ensure the assignee actually has the bandwidth!
